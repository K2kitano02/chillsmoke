## サービス概要

ChillSmoke は、完全禁煙を前提にせず「まずは本数を減らす」ことを支援するアプリです。

喫煙本数を記録すると、設定した基準本数との差から節約額を計算し、欲しいものの購入に使える金額として表示します。
「減らせた本数」を数字だけで終わらせず、金額やウィッシュリストの進捗に変換することで、減煙を続ける理由を作ることを目的にしています。

## 開発背景

実家を出て生活費を意識するようになり、タバコ代の負担を強く感じたことが開発のきっかけです。

禁煙アプリは「完全にやめる」ことを前提にしたものが多く、少しでも吸ってしまうと失敗したように感じやすいと考えました。
そこで、禁煙できない自分を責めるのではなく「昨日より減らせた」「今月これだけ節約できた」と前向きに振り返れるサービスを目指しました。

## 主な機能


| 機能          | 内容                                   |
| ----------- | ------------------------------------ |
| ユーザー登録・ログイン | Devise による認証機能                       |
| 初期設定・再設定    | 目標本数、減煙前の基準本数、1箱の価格、1箱あたりの本数、鬼モードを設定 |
| 今日の喫煙記録     | `+1` 記録、0本記録、日付指定での登録・修正             |
| 喫煙スケジュール    | ルーティン化された喫煙予定を登録し、ボタンひとつで当日ログへ反映     |
| カレンダー表示     | 日ごとの記録をカレンダーで確認。達成・未達・未記録を分けて表示      |
| 節約額の可視化     | 昨日までの累計節約額、今日の節約見込み、使用可能金額を表示        |
| ウィッシュリスト    | 欲しいものを登録し、使用可能金額で購入できるか確認            |
| 購入機能        | 使用可能金額を使ってウィッシュリストを購入済みに変更           |
| 鬼モード        | 目標本数を超えた日は節約額を0円として扱う厳しめのモード         |

## 工夫した点

### 記録を続けやすくする設計

喫煙記録は毎日使う機能のため、ダッシュボードから `+1` で記録できる導線を用意しました。
また、決まった時間に吸うようなルーティン化された喫煙は、喫煙スケジュールとして登録し、ボタンひとつで当日分に反映できるようにしています。

記録し忘れた場合でも後から日付指定で登録・修正できるようにし、継続利用の負担を下げました。

### 未記録と0本を分ける設計

画面表示のために当日のログを自動作成してしまうと、未記録の日が0本扱いになり、実際には記録していないのに達成日として扱われる可能性があります。

そのため、ダッシュボードや日付詳細などの `GET` では喫煙ログを作成せず、保存操作が行われたときだけDBに保存する方針にしています。
これにより、未記録・0本・目標達成・未達を明確に分けられるようにしました。

### 設定変更後も過去の記録が壊れない snapshot 設計

喫煙ログ作成時に、その時点の目標本数、基準本数、価格、1箱あたりの本数、鬼モード設定を snapshot として保存しています。

これにより、後から設定を変更しても、過去日の節約額や達成判定が現在の設定で再計算されて変わってしまうことを防いでいます。

### 重複・同時実行への対策

日次ログは `user_id` と `smoked_on` の組み合わせで一意になるようにし、同じユーザーが同じ日付のログを複数作れない設計にしています。

また、購入処理やスケジュール反映など、同時に実行されると金額や本数がずれる可能性がある処理では、transaction と lock を使って二重購入・二重反映・加算漏れを防ぐようにしました。

### 技術選定の理由

Rails は認証、CRUD、DB設計、バリデーション、ルーティングを一貫して扱いやすく、喫煙記録・設定・購入履歴のようなデータ中心のアプリと相性が良いと考えて選びました。

Rails 8系ではなく 7.2系を採用したのは、学習情報や周辺Gemの互換性が安定しており、卒業制作として実装と検証を進めやすいと判断したためです。

Devise は認証周りの実装ミスがセキュリティリスクにつながりやすいため、自作セッションではなく実績のあるGemを利用しました。
Tailwind CSS は、画面ごとに細かく余白や色を調整しながら、Bootstrapのような既定デザインに寄りすぎず、アプリの雰囲気を作りやすい点を重視しました。

## ローカル環境構築

Docker を使用して起動します。

```bash
docker compose up
```

DB作成・マイグレーションを行います。

```bash
docker compose run --rm web bin/rails db:create db:migrate
```

テストを実行します。

```bash
docker compose run --rm web bin/rails test
```

RuboCop と Brakeman を実行します。

```bash
docker compose run --rm web bundle exec rubocop
docker compose run --rm web bin/brakeman --no-pager
```

### 本番環境で必要な環境変数

Render では、問い合わせメールとパスワード再設定メールを Resend 経由で送信するため、以下の環境変数を設定します。

```text
RESEND_API_KEY=Resendで発行したAPIキー
CONTACT_MAIL_TO=問い合わせを受け取るメールアドレス
CONTACT_MAIL_FROM=Resendで認証済みドメインの送信元メールアドレス
DEVISE_MAILER_FROM=Resendで認証済みドメインのパスワード再設定メール送信元
APP_HOST=chillsmoke.onrender.com
```
